| Значение | Уровень | Описание |
|----------|--------|----------|
| 1 | Низкая | Событие маловероятно и возникает редко |
| 2 | Средняя | Событие возможно при определённых условиях |
| 3 | Высокая | Событие вероятно или уже происходило |

| 3 | Высокое | Потеря данных или критическая недоступность |

# Детальное описание мер реагирования

R-01 — Двойное бронирование парковочного места  
1) Предотвращение (снижение вероятности)

Проверка актуальности парковочного места  
1.1. Перед подтверждением бронирования система проверяет текущий статус парковочного места в базе данных.

Транзакционная обработка запросов  
2.1. Операция бронирования выполняется как единая транзакция PostgreSQL, что предотвращает одновременное изменение данных.

Блокировка записи  
3.1. Во время создания бронирования запись о парковочном месте временно блокируется для других пользователей.

2) Обнаружение (раннее выявление)

Логирование конфликтных операций  
1.1. Система фиксирует все случаи попыток одновременного бронирования одного места.

Оповещение администратора  
2.1. При возникновении конфликта администратор получает уведомление.

3) Реагирование и восстановление

Автоматическая отмена конфликтующей брони  
1.1. Если обнаружено двойное бронирование, система автоматически оставляет только первую успешную операцию.

Возврат денежных средств  
2.1. Пользователь, чья бронь была отменена, получает автоматический возврат средств.

Уведомление пользователей  
3.1. Обе стороны получают уведомление о причине отмены бронирования.

4) Режим деградации

Временная блокировка парковочного места  
1.1. До завершения проверки место может быть переведено в статус “Недоступно”.

5) Пост-инцидент

Анализ причин возникновения конфликта  
1.1. Производится проверка логов, времени запросов и состояния транзакций для выявления причины ошибки.

Корректировка бизнес-логики  
2.1. При повторяющихся конфликтах пересматривается механизм обработки бронирований.

R-02 — Утечка данных банковских карт  
1) Предотвращение (снижение вероятности)

AES-шифрование данных  
1.1. Номера банковских карт хранятся в базе данных только в зашифрованном виде с использованием алгоритма AES.

Использование HTTPS  
2.1. Передача данных между клиентом и сервером осуществляется только через защищённое HTTPS-соединение.

Ограничение доступа  
3.1. Доступ к данным карт имеют только сервисы, участвующие в обработке платежей.

Проверка безопасности API  
4.1. Регулярное тестирование API на наличие уязвимостей.

2) Обнаружение (раннее выявление)

Мониторинг подозрительной активности  
1.1. Система фиксирует аномальные попытки доступа к данным пользователей.

Логирование операций  
2.1. Все действия, связанные с платежами, записываются в журнал событий.

3) Реагирование и восстановление

Блокировка подозрительных запросов  
1.1. При обнаружении подозрительной активности запрос автоматически блокируется.

Оповещение пользователей  
2.1. Пользователь получает уведомление о возможной угрозе безопасности.

4) Режим деградации

Временное отключение оплаты  
1.1. При критической угрозе система может временно отключить проведение платежей.

5) Пост-инцидент

Анализ инцидента  
1.1. Проверка журналов и выявление источника атаки.

Обновление механизмов защиты  
2.1. После инцидента обновляются политики безопасности и механизмы шифрования.

# Критерии готовности реестра рисков

- Для каждого риска назначен владелец  
- Определены причины и последствия  
- Рассчитан приоритет риска  
- Описаны меры предотвращения, обнаружения и реагирования  
- Указан остаточный риск  
- Реестр может быть пересмотрен при изменении архитектуры системы или нагрузки
