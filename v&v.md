# 7. Верификация и валидация (V&V) — ParkPoint

## Обозначения

### Метод проверки:
- **T** — тест (функциональный, интеграционный, нагрузочный)
- **A** — анализ (метрики, SLA, отчёты)
- **I** — инспекция (архитектура, конфигурация, UI, БД)
- **D** — демонстрация (сквозной пользовательский сценарий)

# TC-01 — Въезд автомобиля через LPR

## Цель
Проверка автоматического въезда автомобиля на парковку с использованием системы распознавания номеров (LPR) и открытия шлагбаума при успешной идентификации.

## Предусловия
- Камера распознавания номеров (LPR) активна и подключена к системе ParkPoint  
- Система ParkPoint запущена и доступна  
- Автомобиль зарегистрирован в системе  
- Номер автомобиля присутствует в базе данных  

## Данные
- Номер автомобиля: зарегистрированный в системе  
- Точка въезда: основной въезд парковки  
- Камера: LPR-камера №1  

## Шаги
1. Автомобиль подъезжает к въездному шлагбауму  
2. Камера фиксирует номер автомобиля  
3. Система выполняет распознавание номера (LPR)  
4. Система выполняет проверку номера в базе данных  
5. При совпадении формируется команда на открытие шлагбаума  
6. Система фиксирует событие въезда в журнале событий  

## Ожидаемый результат
- Шлагбаум открывается автоматически  
- Создаётся запись о въезде (номер авто, время, точка въезда)  
- Статус автомобиля изменяется на «на парковке»  
- Событие фиксируется в audit log  

## Постусловия
- Автомобиль находится на территории парковки  
- Сессия парковки создана и активна

# TC-02 — Расчёт стоимости парковки

## Цель
Проверка корректного автоматического расчёта стоимости парковочной сессии.

## Предусловия
- Автомобиль находится на парковке  
- Зафиксировано время въезда  
- Настроен тариф (например: 100 руб/час)  
- Активна парковочная сессия  

## Данные
- Тариф: 100 руб/час  
- Время въезда: фиксированное системой  
- Время выезда: момент завершения сессии  

## Шаги
1. Пользователь завершает парковочную сессию  
2. Система фиксирует время выезда  
3. Вычисляется длительность пребывания  
4. Применяется тарифная модель  
5. Рассчитывается итоговая стоимость  
6. Сумма отображается пользователю  

## Ожидаемый результат
- Стоимость рассчитана корректно  
- Формула: время × тариф  
- Результат соответствует тарифной политике  

## Постусловия
- Сессия переведена в статус «завершена»

# TC-03 — Оплата парковки через приложение

## Цель
Проверка корректной обработки оплаты парковочной сессии через мобильное приложение.

## Предусловия
- Существует завершённая парковочная сессия  
- Пользователь авторизован  
- Подключён платёжный шлюз (mock/имитация)  

## Данные
- Сумма оплаты: рассчитанная системой  
- Способ оплаты: банковская карта (имитация)  

## Шаги
1. Пользователь открывает мобильное приложение  
2. Переходит к активной сессии  
3. Нажимает «Оплатить»  
4. Подтверждает оплату  
5. Система отправляет запрос в платёжный шлюз  
6. Получает ответ о статусе транзакции  

## Ожидаемый результат
- Статус сессии = «Оплачено»  
- Сессия закрыта  
- Пользователь получает уведомление  
- Транзакция фиксируется в журнале событий  

## Постусловия
- Парковочная сессия завершена и оплачена

## Цель

## Предусловия
- Есть доступные парковочные места  
- Пользователь авторизован

2. Выбирает свободное место  
3. Указывает время бронирования  
4. Подтверждает резервирование  
5. Система фиксирует бронь  

## Ожидаемый результат
- Место закреплено за пользователем  
- Статус места = «reserved»  
- Бронь отображается в системе  

## Постусловия
- Место недоступно для других пользователей

# Критерии готовности раздела верификации и валидации

- Каждое требование имеет минимум один метод проверки (T, A, I, D) и связанный артефакт в матрице трассируемости  
- Каждый тест-кейс содержит обязательные разделы: цель, предусловия, шаги, ожидаемые результаты и постусловия  
- Негативные сценарии (LPR-ошибки, отказ оплаты, дублирование въезда, оффлайн-режим) выделены в отдельные TC или вариации  
- Нефункциональные требования проверяются через нагрузочное тестирование, анализ логов, SLA и инспекцию конфигурации  
- Все тесты выполняются в тестовой среде с использованием моков внешних систем (платежи, LPR, датчики)  
- Покрыты все критические end-to-end сценарии (въезд → парковка → оплата → выезд)
