# 4. Архитектура системы

Разрабатываемая система аренды парковочных мест реализована на основе монолитной архитектуры с логическим разделением на модули. Такой подход выбран на этапе проектирования, так как он обеспечивает простоту реализации, снижение сложности интеграции и ускорение разработки.

Несмотря на монолитную структуру, система внутри разделена на функциональные подсистемы, которые соответствуют основным бизнес-процессам предметной области.

---

# 4.1 Декомпозиция системы на компоненты

## 1. Модуль управления пользователями

Отвечает за управление жизненным циклом пользователей системы и контроль доступа.

### Основные функции:
- регистрация пользователей (клиент, владелец, администратор, менеджер)
- аутентификация (email + пароль)
- управление ролями и правами доступа
- управление профилями пользователей
- обработка платёжных данных пользователей

---

## 2. Модуль парковочных зон

Отвечает за управление парковочными зонами, доступными в системе.

### Основные функции:
- создание и редактирование парковочных зон
- хранение информации о локации и адресе
- управление статусом зон (активна / недоступна)
- назначение ответственных менеджеров
- управление медиа-данными (фото, описание)

---

## 3. Модуль парковочных мест

Отвечает за управление отдельными парковочными местами внутри зон.

### Основные функции:
- создание парковочных мест
- привязка мест к парковочной зоне
- назначение владельцев парковочных мест
- управление доступностью мест
- настройка тарифов аренды

---

## 4. Модуль бронирования

Ключевой бизнес-модуль системы, реализующий процесс аренды парковочных мест.

### Основные функции:
- создание бронирования парковочного места
- проверка доступности и конфликтов по времени
- подтверждение и отмена бронирований
- расчёт стоимости аренды
- управление статусами бронирований

---

## 5. Модуль коммуникации

Обеспечивает взаимодействие пользователей внутри системы.

### Основные функции:
- создание чатов между пользователями
- обмен сообщениями между клиентами и владельцами
- хранение истории переписки
- поддержка статусов сообщений

---

## 6. Модуль жалоб

Отвечает за обработку пользовательских жалоб и конфликтных ситуаций.

### Основные функции:
- создание жалоб на пользователей
- рассмотрение жалоб администраторами
- изменение статуса жалобы
- хранение истории обращений

---

## 7. Административный модуль

Отвечает за управление системой и контроль бизнес-процессов.

### Основные функции:
- управление пользователями системы
- модерация парковочных зон и мест
- обработка заявок на создание парковочных зон
- просмотр статистики системы
- управление жалобами пользователей

---

# 4.2 Архитектурные решения и trade-off’ы

Ниже представлены ключевые архитектурные решения, принятые при проектировании системы аренды парковочных мест, а также их преимущества и недостатки.

---

## 4.2 Архитектурные решения

| № | Архитектурное решение | Обоснование | Преимущества | Недостатки |
|---|------------------------|-------------|---------------|-------------|
| 1 | Монолитная архитектура с модульным разделением | Выбрана для упрощения разработки и деплоя на этапе проектирования системы | - Простота разработки и сопровождения<br>- Единый процесс развертывания<br>- Отсутствие сложной инфраструктуры (service discovery, API gateway)<br>- Быстрая отладка и тестирование | - Ограниченная масштабируемость отдельных частей системы<br>- Сложность выделения микросервисов в будущем<br>- Рост связанности при увеличении кода |
| 2 | Логическое разделение на доменные модули | Использование принципов DDD-lite для разделения бизнес-логики по предметным областям | - Улучшенная структурированность кода<br>- Упрощение сопровождения<br>- Снижение связанности компонентов<br>- Более понятная архитектура системы | - Отсутствие физической изоляции модулей<br>- Возможные пересечения логики между доменами |
| 3 | Использование REST API для взаимодействия | REST используется как единый стандарт взаимодействия с внешними клиентами | - Стандартизированный интерфейс<br>- Простота интеграции и тестирования<br>- Поддержка Swagger/OpenAPI<br>- Возможность будущего перехода к микросервисам | - Дополнительный уровень абстракции<br>- Потенциальные накладные расходы при частых вызовах |
| 4 | Использование геолокации (latitude/longitude) | Для обеспечения поиска ближайших парковочных зон | - Удобный пользовательский сценарий “рядом со мной”<br>- Быстрый поиск доступных зон<br>- Возможность масштабирования на карты и навигацию | - Зависимость от точности GPS-данных<br>- Необходимость геоиндексации для производительности |
| 5 | Шифрование чувствительных данных пользователей | Для защиты персональных и платёжных данных | - Повышение уровня безопасности системы<br>- Защита персональных данных пользователей<br>- Снижение риска утечек информации | - Дополнительная нагрузка на систему<br>- Сложность управления ключами шифрования<br>- Усложнение отладки |

---

## Вывод

Принятые архитектурные решения обеспечивают баланс между простотой реализации, безопасностью и возможностью дальнейшего масштабирования системы. Модульный монолит позволяет сохранить низкую сложность разработки, при этом оставляя возможность перехода к распределённой архитектуре в будущем.

Архитектура системы построена на основе монолитного подхода с модульной структурой, что позволяет обеспечить баланс между простотой реализации и функциональной полнотой. В дальнейшем система может быть трансформирована в распределённую архитектуру при увеличении нагрузки.
